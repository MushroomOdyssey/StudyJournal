An absolute link is one that points to any location that uses the following struction for its destination:

scheme://domain/path

The scheme indicates the protocol that the browser must use to request the resource, for example http or https.

The :// is a character pattern that acts as a delimiter between the scheme and the domain. It has no special or inherent meaning. In fact, Tim Berners-Lee (the inventor of the World Wide Web, HTML, the URL system, and HTTP) decided to use this combination of symbols simply because the // was already familiar to him, and the : was more easily human-readable. However, he admits that one small change he would make - if he could do it over again - would have been to drop the // altogether and just use the : after http in URLs!

The path, or more specifically the path to the resource, is kind of a map that points us directly to the content we would like to see. On a static website, this is literally the file structure of the folders and documents stored on the server. On a dynamic website, this is a logical path that is interpreted by the software that is delivering the content.

There may be a huge string of apparent gibberish that is appended to some URLs. These parameters are often key/value pairs separated with the & symbol. Web servers will use this information to perform some voodoo behind the scenes before presenting the resource to you.
You may also sometimes see an anchor to a specific section of the resource, identified in the URL by the #.


Here are some examples of absolute links:
https://www.theodinproject.com/about
https://www.linkedin.com/in/chris-m-gerfen/
https://www.youtube.com/@ambienttechnomixes


But how do we actually "arrive" at these destinations?


If you think of a URL like a set of directions, an absolute link is akin to a street address:

555 Ember Rd.
Dragon Valley, KA
United Kingdoms of New Albion


But while street addresses are formatted from most local (555 Ember Rd.) to least local (United Kingdoms of New Albion) our computer sees it in reverse, starting from the least local position and honing in on the most local end point. And unless it's been there before (or knows someone who has), it'll have to ask for directions:

  • COMPUTER: Oh, great and powerful Oracle of New Albion, have you knowledge of the great realm of Kah Le'Fornia?
    THE ORACLE: Yes, young computer, it is yonder. I shall direct thee to the king.

  • After arriving in Kah Le'Fornia
    COMPUTER: Oh, wise and wondrous King, can you point the way to the dukedom of Dragon Valley?
    THE KING: Yes, brave computer, it is yonder. This honorable knight shall guide thee to the duke.

  • After arriving in Dragon Valley
    COMPUTER: Oh fearsome and pious duke, can you point the way to Ember Road?
    THE DUKE: Yes, foolish computer, it is yonder. Follow the ashen trail. Praise the flame!

  • After arriving in Dragon Valley
    COMPUTER: My quest is complete, and my destination is at hand!


So what does this process look like here in the real world if we wanted to go to www.example.org/page1.html?

  • Well, first a request is sent to the root server, asking, "where can I find .org?".

  • Our computer is redirected by the root server to the .org top-level domain (TLD) server, where it then asks, "where can I find example.org?

  • Our computer is redirected once again by the TLD server to the authoritative name server, where is asks, "what's the IP address of www.example.org?"

  • The authoritative server responds with the appropriate IP address which leads the computer directly to www.example.org, where it sends its final request directly to the server for the page1.html resource.
